본 문서는 핵심 알고리즘 기반 자동 매매 시스템의 End-to-End 안정성을 확보하기 위해, 실시간 데이터 스트림 입력부터 주문 실행까지의 모든 단계에 대한 기술적 검증 항목을 구체화합니다.

## 1. 핵심 데이터 흐름 (Data Flow Map)

시스템은 크게 **데이터 수집 $\rightarrow$ 분석/예측 $\rightarrow$ 신호 생성 $\rightarrow$ 주문 실행**의 네 단계로 구성됩니다. 각 단계별 입력, 처리 로직, 출력 및 연관 API를 명시합니다.

| 단계 | 모듈/처리 내용 | 주요 입력 데이터 (Input) | 핵심 처리 로직 (Process) | 주요 출력 데이터 (Output) | 연관 시스템/API |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Stage 1: Data Ingestion** | 실시간 데이터 스트리밍 수집 | 키움 API 실시간 시세 데이터 | 데이터 정규화 및 시간 동기화 (Timestamp Sync) | 정제된 시세 데이터 스트림 | Kiwoom API Stream |
| **Stage 2: Prediction Engine** | 상승/우상향/급등 예측 알고리즘 실행 | 정제된 시세 데이터 | 핵심 지표(예측 모델) 계산 및 결과 산출 (Scoring) | 예측 신호 (Signal: Buy/Hold/Sell) 및 신뢰도 점수 | 내부 ML/알고리즘 모듈 |
| **Stage 3: Signal Generation** | 매매 결정 로직 적용 | 예측 신호, 현재 포트폴리오 데이터 | 위험 관리 규칙(Risk Management) 및 조건 필터링 | 최종 주문 생성 요청 (Order Request Object) | 시스템 내부 로직 |
| **Stage 4: Order Execution** | 자동 매매 실행 | 최종 주문 생성 요청 | API 인증 및 주문 전송 | 주문 결과 (Execution Result) | Kiwoom API Order Execution |

## 2. 잠재적 오류 발생 시나리오 (Error Scenarios & Edge Cases)

각 단계에서 발생 가능한 모든 비정상 상태(Error State)와 예외 상황을 정의하고, 시스템의 회복 메커니즘(Recovery Mechanism)이 작동하는지 검증합니다.

### A. 데이터 수집 단계 (Stage 1: Data Ingestion) 오류 시나리오

| ID | 시나리오 설명 | 예상되는 오류 유형 | 기대되는 시스템 반응 및 회복 |
| :--- | :--- | :--- | :--- |
| **E1.1** | 키움 API 연결 끊김 또는 타임아웃 발생 | Connection Timeout, API Error (HTTP 5xx) | 자동 재연결(Retry Logic). 데이터 누락 최소화 확인. |
| **E1.2** | 실시간 데이터 스트림의 순서 불일치 | Out-of-Order Data Detected | 시간 동기화 메커니즘 작동 여부 검증. 시계열 데이터 정합성 확인. |
| **E1.3** | API 응답에 Null 값 또는 잘못된 형식 데이터 수신 | Data Format Error, Null Pointer Exception | 입력값 필터링(Sanitization). 해당 데이터 포인트 무시 및 로그 기록. |

### B. 예측/분석 단계 (Stage 2: Prediction Engine) 오류 시나리오

| ID | 시나리오 설명 | 예상되는 오류 유형 | 기대되는 시스템 반응 및 회복 |
| :--- | :--- | :--- | :--- |
| **E2.1** | 알고리즘 입력 데이터 부족 (데이터 스트림 끊김) | Insufficient Data for Calculation | 예측 신호 대신 '대기(Hold)' 상태로 전환. 경고 알림 발생. |
| **E2.2** | 예측 모델의 비정상적인 출력값 (Outlier Prediction) | Model Output Out-of-Range | 예측 점수(Score)를 시스템 안정성 KPI에 따라 필터링하고, 전문가 개입이 필요한 플래그 설정. |
| **E2.3** | 내부 계산 과정에서 메모리 오류 발생 | Memory Allocation Error | 프로세스 강제 종료 방지 및 롤백 메커니즘 작동 확인. |

### C. 주문 실행 단계 (Stage 4: Order Execution) 오류 시나리오

| ID | 시나리오 설명 | 예상되는 오류 유형 | 기대되는 시스템 반응 및 회복 |
| :--- | :--- | :--- | :--- |
| **E4.1** | 주문 전송 중 네트워크 오류 발생 | Network Failure, Timeout | 재시도(Retry) 로직 실행 후 최종 실패 시 즉시 시스템 정지(Fail-Safe). |
| **E4.2** | 주문 조건 불일치 (예: 잔고 부족, 가격 미달) | Order Rejection Error | 주문 거부 사유를 상세히 기록하고, 다음 신호 생성 단계로 피드백 전달. |
| **E4.3** | API 인증 만료 또는 권한 오류 | Authentication/Authorization Error | 즉시 시스템 접근 차단 및 관리자 알림 발생. |

## 3. 기술적 검증 항목 (Technical Verification Checklist)

각 시나리오에 대해 다음의 항목들이 충족되는지 확인합니다.

1.  **데이터 무결성 점검:** 모든 단계에서 데이터가 손실되거나 왜곡되지 않고 정확히 전달되었는가? ($\text{Data Integrity Check}$)
2.  **지연 시간 측정:** 각 단계(Stage 1 $\rightarrow$ Stage 4)별 평균 지연 시간을 측정하고, 목표 Latency ($\pm 5\%$)를 초과하지 않는가? ($\text{Latency Monitoring}$)
3.  **오류 로그 기록 점검:** 모든 오류 발생 시, 오류 유형, 발생 시점, 시스템 상태(State)가 상세히 로그 파일에 기록되었는가? ($\text{Logging Verification}$)
4.  **회복 메커니즘 검증:** E1.1, E4.1과 같은 외부/네트워크 오류 발생 시, 시스템이 설정된 회복 절차(Retry, Fail-Safe)를 정확히 수행하는가? ($\text{Recovery Mechanism Test}$)
5.  **KPI 연동 확인:** 발생한 모든 오류 및 지연 시간이 최종 KPI (오류율 $\le 0.5\%$, Latency $\pm 5\%$)에 미치는 영향을 실시간으로 모니터링할 수 있는 대시보드가 준비되었는가? ($\text{KPI Dashboard Linkage}$)

### 3. 기술적 검증 항목 (Technical Verification Checklist)

시스템 안정성 KPI(오류율 0.5% 미만, 지연 시간 ±5%) 달성을 위해 반드시 테스트해야 할 구체적인 항목 리스트입니다.

#### A. 정확성 검증 (Accuracy Verification)
1.  **알고리즘 일치도:** 실제 데이터 기반 예측 결과와 알고리즘이 도출한 신호 간의 상관관계 분석 (F1 Score 또는 유사 지표 사용).
2.  **Edge Case 테스트:** 시장 급변기(Volatility Spike), 장시간 횡보장, 극단적인 변동성 구간에서 시스템이 안정적으로 작동하는지 검증.
3.  **데이터 무결성:** API로부터 수신된 데이터의 양과 질이 실제 거래소 데이터와 일치하는지 100% 매칭 테스트.

#### B. 안정성 및 속도 검증 (Stability & Latency Verification)
1.  **Latency 측정:** Step 1부터 Step 5까지 각 단계별 처리 시간을 측정하여 **평균 지연 시간(Latency)**을 산출하고, 목표치 $\pm 5\%$를 준수하는지 확인.
2.  **오류율 측정:** 모든 오류 시나리오(E-1 ~ E-6) 발생 시점의 기록을 통해 실제 시스템 오류율을 계산하여 **0.5% 미만** 달성 여부를 검증.
3.  **스트리밍 안정성:** 데이터 스트림이 끊기지 않고 지속적으로 수신되는지, 단절 시 복구 메커니즘이 즉각 작동하는지 테스트.

#### C. 통합 및 보안 검증 (Integration & Security Verification)
1.  **권한 분리 확인:** 매매 실행 권한과 데이터 조회 권한이 논리적으로 완벽히 분리되어 있으며, 한쪽의 오류가 다른 쪽에 영향을 주지 않음을 감사(Audit) 로그를 통해 확인.
2.  **역추적 검증:** 최종 주문 결과(Step 5)가 최초 입력 데이터(Step 1)와 알고리즘 계산(Step 2)을 거쳐 논리적으로 완벽하게 도출되었는지 역추적 테스트 수행.

## 1. 시스템 안정성 KPI 및 목표 정의 (KPI Definition)

Phase 1의 목표는 알고리즘의 **기능적 정확성**과 **시스템 안정성**을 검증하는 데 중점을 둡니다.

| 지표 | 목표 기준 | 측정 대상 |
| :--- | :--- | :--- |
| **오류율 (Error Rate)** | $\le 0.5\%$ | API 호출 실패, 데이터 처리 오류, 매매 실행 실패 등 모든 예외 발생 건수 |
| **평균 지연 시간 (Latency)** | $\pm 5\%$ | 실시간 데이터 수신부터 매매 명령 전송까지의 평균 소요 시간 |
| **데이터 무결성** | $100\%$ | 입력된 API 데이터와 시스템 내부 계산 결과 간의 일치 여부 |

## 2. 정의된 오류 시나리오 (Error Scenarios Definition)

시스템이 예측하지 못한 상황이나 외부 환경 변화에 대응하여 안정성을 검증하기 위해 다음의 주요 오류 시나리오를 테스트 케이스에 포함합니다.

1.  **API 연결 오류:** 키움 API 서버와의 연결이 일시적으로 끊겼을 때의 재연결 및 에러 처리 (4xx, 5xx 응답).
2.  **데이터 누락/오류:** 실시간 데이터 스트리밍 중 특정 지표(예: 호가, 거래량)가 누락되거나 비정상적인 값(Null, 비현실적 수치)이 들어왔을 때의 처리.
3.  **알고리즘 예외:** 예측 알고리즘이 극단적인 조건(예: 급변하는 시장 상황)에서 비정상적인 신호를 출력할 때의 안전 장치 작동 여부 (Stop-Loss/Kill Switch 발동).
4.  **매매 실행 실패:** 주문 전송 시 발생할 수 있는 네트워크 지연 또는 서버 측 거부 오류에 대한 재시도 및 로그 기록 검증.

## 3. Phase 1 테스트 코드 초안 (Python Mocking & Validation)

실제 API 연동 대신 모의 데이터(Mock Data)를 사용하여 알고리즘 로직과 에러 핸들링 모듈의 안정성을 우선 검증하는 코드를 제시합니다.

**파일명:** `test_algorithm_phase1.py`

```python
import time
import random
from typing import Dict, Any

# --- 1. 핵심 알고리즘 (Mock) ---
def calculate_prediction(data: Dict[str, float]) -> str:
    """실제 예측 로직을 모의하는 함수."""
    if data.get('price') is None or data.get('volume') == 0:
        raise ValueError("데이터 누락 오류: 가격 또는 거래량 정보가 없습니다.")
    
    # 예시: 단순 임계값 기반 예측 (실제 알고리즘 대체)
    if data['price'] > 1000 and data['volume'] > 50000:
        return "STRONG_BUY"
    elif data['price'] < 500:
        return "STRONG_SELL"
    else:
        return "HOLD"

# --- 2. API 연동 및 에러 핸들링 모듈 (Mock) ---
def mock_kium_api_call(symbol: str, request_type: str) -> Dict[str, Any]:
    """API 호출을 모의하며 의도적으로 오류를 발생시킴."""
    
    # 1. API 연결 오류 시나리오 (5% 확률로 실패)
    if random.random() < 0.05:
        raise ConnectionError(f"API 서버 연결 실패: {symbol}")
        
    # 2. 데이터 누락/오류 시나리오 (10% 확률로 Null 값 반환)
    if random.random() < 0.10:
        return {"error": "DATA_MISSING", "status": "ERROR"}

    # 성공적인 데이터 반환 시뮬레이션
    mock_data = {
        'symbol': symbol,
        'price': round(random.uniform(400, 2000), 2),
        'volume': random.randint(10000, 100000) if request_type == 'realtime' else 0,
        'timestamp': time.time()
    }
    return mock_data

def execute_trade_order(symbol: str, order_type: str, amount: int) -> Dict[str, Any]:
    """매매 주문 실행을 모의하는 함수."""
    if random.random() < 0.03:  # 매매 실패 시나리오 (3% 확률)
        raise TimeoutError(f"주문 전송 시간 초과: {symbol}")
    
    print(f"[TRADE_SUCCESS] {order_type} 주문 성공: {symbol}, 수량: {amount}")
    return {"status": "SUCCESS", "order_id": f"ORD-{int(time.time())}"}

# --- 3. 통합 테스트 로직 (Integration Test Flow) ---
def run_phase1_test_suite(symbol: str, num_iterations: int = 100):
    """전체 시스템의 안정성을 검증하는 메인 테스트 루틴."""
    print(f"\n--- Phase 1 Test Start for {symbol} ({num_iterations} iterations) ---")
    successful_trades = 0
    failed_calls = 0
    latency_log = []

    for i in range(num_iterations):
        try:
            # Step 1: 실시간 데이터 수신 (Latency 측정 시작점)
            start_time = time.time()
            realtime_data = mock_kium_api_call(symbol, 'realtime')
            latency = time.time() - start_time

            # Step 2: 알고리즘 예측
            prediction = calculate_prediction(realtime_data)

            # Step 3: 매매 결정 및 실행 (에러 시나리오 포함)
            if prediction == "STRONG_BUY":
                result = execute_trade_order(symbol, 'BUY', 10)
                successful_trades += 1
            else:
                # HOLD 또는 STRONG_SELL의 경우 주문 생략 또는 보류
                pass

            latency_log.append(latency)

        except (ConnectionError, TimeoutError, ValueError) as e:
            failed_calls += 1
            error_type = type(e).__name__
            print(f"[ERROR DETECTED] Iteration {i+1}: {error_type} - {e}")
            # 오류 발생 시 시스템이 안전하게 대기하도록 로직을 삽입 (Kill Switch 역할)

    # --- 4. 결과 분석 및 보고 (KPI 측정) ---
    if num_iterations > 0:
        avg_latency = sum(latency_log) / len(latency_log) if latency_log else 0
        error_rate = (failed_calls / num_iterations) * 100

        print("\n=========================================")
        print("✅ Phase 1 Test Summary Report")
        print("=========================================")
        print(f"총 반복 횟수: {num_iterations}")
        print(f"성공적인 거래 실행: {successful_trades}회")
        print(f"실패한 API 호출/처리: {failed_calls}회 (오류율: {error_rate:.2f}%)")
        print(f"평균 지연 시간 (Latency): {avg_latency:.4f} 초")

        # KPI 검증 로직
        if error_rate <= 0.5 and abs(avg_latency) <= 5.0:
            print("\n🌟 ✅ 시스템 안정성 목표 달성: 모든 핵심 KPI를 통과했습니다.")
        else:
            print("\n🚨 ⚠️ 시스템 안정성 경고: 일부 KPI 기준을 충족하지 못했습니다. 추가 디버깅이 필요합니다.")

# --- 실행 예시 ---
if __name__ == "__main__":
    TEST_SYMBOL = "KRX:005930" # 테스트할 종목 코드
    run_phase1_test_suite(TEST_SYMBOL, num_iterations=200)
```
