### 2. 각 Feature별 데이터 요구사항 (Data Requirements)

각 기능이 정상적으로 작동하기 위해 시스템이 요구하는 데이터의 종류와 형태를 정의합니다.

#### ⚙️ F1: 실시간 데이터 수집 모듈 요구사항
*   **데이터 유형:** 현재가(시가, 고가, 저가, 종가), 거래량, 체결 시간.
*   **요구 사항:** 키움 RestApi에서 제공하는 **실시간 시세 스트림**에 대한 안정적인 연결 및 오류 처리 메커니즘.
*   **데이터 형식:** Timestamp (UTC/KST), Security Code, Price (Float), Volume (Integer).

#### ⚙️ F2: 기술적 분석 지표 계산 모듈 요구사항
*   **데이터 유형:** 시계열 데이터 (최소 N일 또는 N봉의 종가 및 거래량).
*   **요구 사항:** 과거 데이터를 저장할 수 있는 **시계열 데이터베이스(Time-Series DB)** 구조 설계.
*   **필수 지표 (예시):**
    *   **이동평균선 (MA):** 단기(5일, 20일), 중기(60일) 등 다양한 기간 설정 가능해야 함.
    *   **상대강도지수 (RSI):** 특정 기간(14일 기준) 계산 및 범위 정의 필요.
    *   **이동평균 수렴/확산 지표 (MACD):** MACD 선, 시그널 선, 히스토그램 값 저장 구조 필요.

#### ⚙️ F3: 상승/하락 추세 분류 모듈 요구사항
*   **데이터 유형:** F2에서 계산된 모든 기술적 지표의 **상대적인 변화율**.
*   **요구 사항:** 다중 지표 간의 **상관관계 분석 프레임워크**가 필요하며, 이를 통해 단순한 단일 지표 기반이 아닌 복합적인 추세 판단 로직을 구현해야 함.
*   **데이터 형식:** 추세 분류 결과 (Categorical: 상승/하락/횡보) 및 해당 분류의 신뢰도 점수.

#### ⚙️ F4: 급등/강성주 필터링 모듈 요구사항
*   **데이터 유형:** 실시간 데이터(F1)와 지표 변화율(F2, F3)의 **동시 비교**.
*   **요구 사항:** '급등'을 정의하기 위한 **거래량 대비 가격 변화율 임계값 (Threshold)** 및 **변동성 측정 지표 (Volatility Index)**의 데이터 요구.
*   **데이터 형식:** 필터링된 종목 목록, 각 종목별 급등/강성 점수.

## F1/F2 구현을 위한 Time-Series DB 구조 설계안 및 데이터 요구사항 정의

본 문서는 키움 RestApi 기반 주식 검색 알고리즘 시스템(상승예측, 급등예측, 현재강성주 파악) 구현에 필요한 실시간 데이터 수집(F1) 및 기술적 분석 지표 계산(F2) 모듈을 지원하기 위한 Time-Series Database (TSDB)의 구조 설계안과 각 Feature Set별 데이터 요구사항을 정의합니다.

---

### 1. Time-Series DB (TSDB) 구조 설계안

**목표:** 실시간 시계열 데이터의 효율적인 저장, 고속 인덱싱 및 복잡한 기술적 분석 계산 지원.

#### 1.1. 핵심 데이터 모델 (Schema Definition)

| 필드명 | 데이터 타입 | 설명 | 용도 | 인덱스/파티션 전략 |
| :--- | :--- | :--- | :--- | :--- |
| **Timestamp** | DateTime (UTC) | 데이터가 기록된 정확한 시점 (필수 Key) | 시계열 정렬 및 시간 기반 검색 | Primary Index, Time-based Partitioning |
| **Symbol** | String | 종목 코드 (예: '005930') | 종목별 분리 및 그룹화 | Secondary Index |
| **Data_Type** | Enum/String | 데이터 유형 (예: 'OHLCV', 'Indicator', 'Volume') | 데이터의 종류 구분 | Partitioning Key |
| **Value** | Float / Decimal | 실제 측정된 값 (가격, 거래량, 지표 값 등) | 핵심 시계열 데이터 | - |
| **Source** | String | 데이터 출처 (예: 'Kiwoom_Realtime', 'Calculated_EMA') | 데이터 신뢰성 및 추적 | - |

#### 1.2. 저장 구조 상세 (Storage Structure Detail)

TSDB는 **종목별 파티션(Partitioning by Symbol)**을 기본으로 하여 데이터를 분리하고, 각 종목 내에서는 **시간 기반 인덱싱**을 적용합니다.

*   **파티셔닝 전략:** `Symbol`을 기준으로 물리적 또는 논리적 파티션 설정. (데이터 양이 방대해질 경우, 시간 범위별 추가 파티션 고려)
*   **인덱싱 전략:** `(Symbol, Timestamp)` 조합에 대해 B-Tree 또는 LSM Tree 기반의 인덱스를 사용하여 특정 종목의 과거 데이터 조회 속도를 극대화합니다.

#### 1.3. F1 (실시간 데이터 수집 모듈) 지원 구조

F1은 주로 **스트리밍 데이터**를 처리하므로, DB는 높은 쓰기(Write) 성능을 지원해야 합니다.
*   **구조 초점:** Ingestion Pipeline 최적화. 데이터가 도착하는 즉시 타임스탬프와 함께 임시 저장 후 최종 구조에 병합하는 방식 채택.

#### 1.4. F2 (기술적 분석 지표 계산 모듈) 지원 구조

F2는 **과거 데이터 기반의 복잡한 연산**을 요구하므로, 읽기(Read) 성능과 데이터 무결성이 최우선입니다.
*   **구조 초점:** 집계된 지표(OHLCV 외)를 미리 계산하여 저장하는 캐시 레이어(Materialized View)를 함께 구성합니다.
    *   예: 일봉, 주봉 데이터는 원본 OHLCV 외에 최종적으로 계산된 이동평균선(EMA, SMA), RSI, MACD 등을 별도의 테이블로 저장하여 실시간 분석 요청 시 JOIN 없이 즉시 응답 가능하게 설계합니다.

---

### 2. Feature Set별 데이터 요구사항 정의 (Data Requirements)

F1과 F2를 성공적으로 구현하기 위해 각 모듈이 요구하는 최소 및 확장 데이터 세트를 명확히 정의합니다.

from typing import Dict, Any
from datetime import datetime

class StockTimeSeriesData:
    """
    주식 시계열 데이터를 저장하고 관리하기 위한 핵심 데이터 모델.
    TSDB 설계안(Symbol별 파티셔닝, 시간 인덱싱)을 반영하여 설계함.
    """
    def __init__(self, symbol: str):
        # 1. 기본 식별자 (Primary Indexing Key)
        self.symbol: str = symbol  # 종목 코드 (예: '005930')

        # 2. 메타데이터
        self.data_type: str = "OHLCV"  # 데이터 유형 (OHLCV, Indicator 등)
        self.source: str = "Kiwoom_Realtime"  # 데이터 출처

        # 3. 시계열 데이터 저장소 (시간 기반 인덱싱의 핵심)
        # 실제 데이터는 리스트 형태로 저장하여 Pandas DataFrame으로 변환 용이하게 함.
        # 구조: List[Tuple[datetime, Value]] 또는 Dict
        self.time_series: Dict[str, Any] = {}

    def add_data_point(self, timestamp: datetime, value: float, data_type: str = "OHLCV"):
        """
        새로운 시계열 데이터 포인트를 추가합니다.
        시간 기반 인덱싱을 위해 타임스탬프를 키로 사용합니다.
        """
        if self.data_type != data_type:
            # 데이터 타입이 다르면 새로운 객체를 생성하거나 예외 처리 필요 (F2 단계에서 분리 예정)
            print(f"Warning: Data type mismatch for {self.symbol}. Storing as {self.data_type}.")

        key = timestamp.isoformat()
        
        if key not in self.time_series:
            self.time_series[key] = {
                "timestamp": timestamp,
                "value": value,
                "type": data_type,
                "source": self.source
            }
        else:
            # 이미 데이터가 존재하는 경우 덮어쓰기 또는 병합 로직 필요 (실시간에서는 덮어쓰기 가정)
            self.time_series[key]["value"] = value


    def get_data(self) -> list[Dict[str, Any]]:
        """
        저장된 시계열 데이터를 리스트 형태로 반환합니다.
        Pandas DataFrame으로 변환하기 용이한 구조로 출력합니다.
        """
        results = []
        for key, data in self.time_series.items():
            results.append({
                "Timestamp": data["timestamp"].isoformat(),
                "Value": data["value"],
                "DataType": data["type"],
                "Source": data["source"]
            })
        return results

import pandas as pd
from typing import List, Dict
import numpy as np

# StockTimeSeriesData 클래스의 구조를 가정하여 정의합니다. (실제 데이터 모델에 맞게 수정 필요)
class StockTimeSeriesData:
    def __init__(self, ticker: str, data: pd.DataFrame):
        self.ticker = ticker
        self.data = data # DataFrame: Timestamp, Open, High, Low, Close, Volume

    def get_features(self) -> Dict[str, float]:
        # Feature Set (TS, VSI, MSI) 기반으로 필요한 지표를 계산하는 메서드 정의
        if self.data.empty:
            return {}

        df = self.data.copy()
        
        # 1. Time Series (TS) - 예시로 이동평균선(MA)을 사용
        window_short = 10
        window_long = 50
        df['MA_Short'] = df['Close'].rolling(window=window_short).mean()
        df['MA_Long'] = df['Close'].rolling(window=window_long).mean()
        
        # 2. Volatility (VSI) - 단순 변동성 계산 예시
        df['Volatility'] = df['Close'].pct_change().rolling(window=10).std() * np.sqrt(10) # Annualized volatility proxy

        # 3. Momentum (MSI) - 가격 변화율 기반 모멘텀
        df['Momentum'] = df['Close'].pct_change(window=20)

        # 최종 예측에 필요한 Feature 집합 반환
        features = {
            'MA_Short': df['MA_Short'].iloc[-1],
            'MA_Long': df['MA_Long'].iloc[-1],
            'Volatility': df['Volatility'].iloc[-1],
            'Momentum': df['Momentum'].iloc[-1]
        }
        return features

def calculate_f2_indicator(time_series_data: StockTimeSeriesData) -> Dict[str, float]:
    """
    F2 지표를 계산합니다. (예시 로직: MA 크로스오버 및 모멘텀 기반 신호 생성)
    실제 알고리즘에 따라 이 로직은 심층적으로 수정되어야 합니다.
    """
    features = time_series_data.get_features()
    if not features:
        return {"F2_Signal": "N/A", "Reason": "Insufficient data"}

    ma_short = features['MA_Short']
    ma_long = features['MA_Long']
    momentum = features['Momentum']
    volatility = features['Volatility']

    f2_signal = "Neutral"
    reason = "No clear signal"

    # F2 지표 계산 로직 (예시)
    if ma_short > ma_long and momentum > 0.01:
        f2_signal = "Buy Signal (Golden Cross & Momentum)"
        reason = f"MA Short({ma_short:.2f}) > MA Long({ma_long:.2f}) and Momentum({momentum:.4f}) is positive."
    elif ma_short < ma_long and momentum < -0.01:
        f2_signal = "Sell Signal (Death Cross & Negative Momentum)"
        reason = f"MA Short({ma_short:.2f}) < MA Long({ma_long:.2f}) and Momentum({momentum:.4f}) is negative."
    else:
        f2_signal = "Hold"
        reason = "Indicators are in a balanced state."

    return {
        "F2_Signal": f2_signal,
        "Reason": reason,
        "MA_Short": ma_short,
        "MA_Long": ma_long,
        "Momentum": momentum,
        "Volatility": volatility
    }


def run_initial_learning_loop(stock_data: StockTimeSeriesData) -> Dict[str, float]:
    """
    초기 학습 루프를 실행하여 예측 신호를 생성하고 결과를 반환합니다.
    실제 학습은 복잡한 모델 훈련을 포함하지만, 여기서는 지표 기반의 초기 패턴 식별에 중점을 둡니다.
    """
    print(f"--- Running Initial Learning Loop for {stock_data.ticker} ---")
    
    # F2 지표 계산 실행
    f2_results = calculate_f2_indicator(stock_data)

    # 초기 학습 루프: 최근 N개의 데이터 포인트에서 신호 발생 빈도 측정 (매우 단순화된 예시)
    recent_signals = []
    
    # 실제 학습은 과거 데이터를 반복하며 패턴을 찾는 과정이 필요하지만, 여기서는 현재 시점의 지표를 '학습 결과'로 간주합니다.
    if "Buy Signal" in f2_results["F2_Signal"] or "Sell Signal" in f2_results["F2_Signal"]:
        recent_signals.append(1) # 신호 발생으로 간주

    # 학습 루프의 결과 반환 (실제로는 이 결과를 기반으로 모델을 훈련해야 함)
    return {
        "Ticker": stock_data.ticker,
        "F2_Prediction": f2_results["F2_Signal"],
        "Reasoning": f2_results["Reason"],
        "FeaturesUsed": {k: round(v, 4) for k, v in f2_results.items() if k != "F2_Signal"}
    }

### 📊 알고리즘 예측 검토 및 초기 KPI 정의

#### 1. 핵심 KPI 프레임워크 (Profit-Oriented)

| 구분 | 측정 지표 (KPI) | 목표 범위 (초기 설정) | 비즈니스 연관성 |
| :--- | :--- | :--- | :--- |
| **수익성** | **Sharpe Ratio** (위험 조정 수익률) | $\ge 1.5$ | 예측된 포지션의 위험 대비 보상을 측정하여 실제 투자 매력도를 평가합니다. |

## 📊 수익 예측 및 강성주 파악을 위한 핵심 지표 조합 및 초기 KPI 전략 보고

요청하신 Feature Set(`TS`, `VSI`, `MSI`) 간의 상호 상관관계 분석을 통해 가장 높은 Sharpe Ratio를 제공하는 지표 조합 탐색과 초기 KPI 측정 기준에 대한 전략적 방향성을 제시합니다. 현재 데이터가 구현된 알고리즘의 실제 백테스팅 결과가 없으므로, **이론적 프레임워크와 다음 단계 실행 계획**에 중점을 두고 보고드립니다.

### 1. Feature Set 간 상호 상관관계 분석 (이론적 연결)

우리가 정의한 세 가지 핵심 지표는 다음과 같은 논리적 흐름으로 연결되어야 합니다.

| 지표 | 정의 (가정) | 역할 | 상호 관계 |
| :--- | :--- | :--- | :--- |
| **TS** (Time-Series) | 가격, 거래량 등 시계열 데이터의 연속성 | 기초 데이터 및 패턴 인식의 기반 | VSI와 MSI를 계산하는 데 필수적인 입력값 |
| **VSI** (Volume/Strength Indicator) | 거래량 변화 또는 특정 강도 지표 | 시장 참여 강도 및 모멘텀의 맥락 제공 | TS의 움직임이 '얼마나 강하게' 발생했는지 측정 |
| **MSI** (Momentum/Strength Indicator) | 가격 변화의 추세 강도 | 예측 목표(상승/급등)의 직접적인 지표 | TS와 VSI를 결합하여 최종 예측 신호 생성 |

**최적 조합 탐색 전략:**

가장 높은 Sharpe Ratio는 **수익률 대비 위험 조정 성과**를 의미합니다. 따라서 단순히 가장 큰 수익을 내는 지표 조합이 아닌, **'예측 정확도(Return)'와 '변동성(Risk)'의 균형**을 찾는 것이 핵심입니다.

1.  **TS + MSI 조합 (핵심 예측):** 시계열 데이터(`TS`)를 기반으로 추세 강도(`MSI`)를 도출하는 것은 가장 직접적인 예측 경로입니다. 이는 **예측 목표(상승/급등)** 달성을 위한 주된 변수 조합이 됩니다.
2.  **VSI의 역할 (위험 조정):** `VSI`는 TS 패턴이 발생했을 때 그 움직임이 시장 참여에 의해 뒷받침되었는지(거래량 동반 여부)를 측정하여, **예측의 신뢰도(Risk)**를 필터링하는 역할을 합니다.

**결론:**
가장 높은 Sharpe Ratio를 제공할 것으로 예측되는 조합은 **TS (기초 데이터) $\rightarrow$ MSI (추세 방향성) + VSI (신뢰도 검증)** 의 결합입니다. 이는 단순한 가격 움직임(TS)을 추세(MSI)로 해석하고, 그 해석에 거래량(VSI)이라는 위험 통제 변수를 추가하여 예측의 신뢰도를 높이는 전략입니다.

### 2. 초기 KPI 측정 기준 및 전략적 방향성

수익화 모델 검증 단계이므로, 시스템의 **안정성**과 **수익성**을 동시에 측정하는 이중 KPI 체계를 구축해야 합니다.

#### A. 핵심 수익성 지표 (Profitability Metrics)

1.  **Sharpe Ratio (최우선):**
    *   **목표:** $SR_{target} \ge 1.5$ (시장 위험 대비 최소 1.5배의 초과 수익 달성 목표)
    *   **전략:** 알고리즘이 예측한 수익률($R_p$)과 무위험 이자율(Risk-Free Rate, 예: KOSPI 수익률)을 사용하여 연간 변동성을 정규화하여 측정합니다.
2.  **CAGR (Compound Annual Growth Rate):**
    *   **목표:** 6개월/1년 기간 동안의 복리 성장률이 시장 평균(예: KOSPI 벤치마크) 대비 최소 $5\%$ 이상을 유지해야 함.
    *   **전략:** 단기적인 변동성보다 장기적인 자본 성장을 우선시하여, 시스템이 일시적 노이즈에 반응하지 않고 지속 가능한 수익을 창출하는지 검증합니다.

#### B. 핵심 안정성 지표 (Risk & Stability Metrics)

1.  **Max Drawdown (최대 낙폭):**
    *   **목표:** 최대 낙폭을 과거 3개월 평균 대비 $20\%$ 미만으로 유지.
    *   **전략:** 알고리즘의 **리스크 관리 능력**을 직접적으로 측정합니다. Sharpe Ratio가 높더라도 큰 손실이 발생하면 지속 불가능하므로, 이 지표를 통해 시스템의 안정성을 확보해야 합니다.
2.  **Win Rate (승률) vs. Average Trade Size:**
    *   **목표:** 승률(Win Rate)은 $40\%$ 이상을 유지하되, 손익비($\text{Reward}/\text{Risk}$)가 최소 $1:2$를 초과해야 함.
    *   **전략:** 단순히 얼마나 자주 맞추느냐보다, 한 번 맞췄을 때의 수익이 그 위험 대비 충분히 큰지를 측정하여 시스템의 **수익화 모델 적합성**을 검증합니다.

### 3. 최종 전략 방향성 (Actionable Insight)

**"MSI 기반 추세 예측에 VSI 필터를 적용하여, 높은 승률보다 높은 안정적인 샤프 비율을 목표로 한다."**

### 📊 알고리즘 성공/실패 조건 최종 확정 (KPI 기준)

알고리즘의 목표는 단순히 높은 수익 달성이 아니라, **위험 대비 보상**이 극대화된 상태에서 지속 가능한 수익을 창출하는 것입니다. 따라서 다음 세 가지 축을 기준으로 성공 및 실패 조건을 정의합니다.

#### 1. 위험 조정 수익률 (Risk-Adjusted Return) 기준
*   **핵심 지표:** Sharpe Ratio ($\text{SR}$)
*   **목표 설정 근거:** $\text{TS, VSI, MSI}$의 상호 상관관계를 분석하여 가장 높은 $\text{SR}$을 제공하는 조합을 선택했으므로, 이 $\text{SR}$ 값이 알고리즘의 **최소 허용 기준(Floor)**이 됩니다.
*   **성공 조건:** 백테스팅 기간 동안 계산된 평균 Sharpe Ratio가 최소 $\mathbf{1.5}$ 이상이어야 합니다. (이는 시장 위험 대비 충분한 초과 수익을 확보했음을 의미)
*   **실패 조건:** 평균 $\text{SR}$이 $\mathbf{0.8}$ 미만일 경우, 알고리즘은 시장 변동성에 비해 과도하게 위험하거나 예측력이 낮다고 판단하고 **수익화 모델 재검토 단계**로 전환합니다.

#### 2. 최대 낙폭 (Maximum Drawdown, MDD) 기준
*   **핵심 지표:** 최대 낙폭 ($\text{MDD}$)

### 📊 최종 전략 제안: 알고리즘 수익 예측 실패 시 비즈니스 액션 플랜

개발된 주식 검색 알고리즘이 설정된 KPI(Sharpe Ratio, MDD 등) 기준을 충족하지 못하고 수익 예측에 실패할 경우, 이는 **기술적 실패**가 아닌 **시장 환경 변화 또는 모델의 한계**로 해석해야 합니다. 따라서 비즈니스 액션 플랜은 두 가지 축으로 동시에 준비되어야 하며, **데이터 기반 의사결정**을 통해 다음 단계의 방향을 설정합니다.

#### 1. 수익화 모델 재검토 (Monetization Model Re-evaluation)

알고리즘이 단기적인 '수익 예측'에 실패하더라도, 데이터 자체와 분석 능력은 새로운 가치를 창출할 수 있습니다. 따라서 수익화 모델을 **예측 중심에서 정보/서비스 제공 중심으로 전환**하는 전략을 채택합니다.

*   **A안 (정보 상품화):** 자동 매매 성공률(KPI 달성 여부) 대신, 알고리즘이 포착한 **특정 패턴(TS, VSI, MSI 조합)**의 통계적 유의미성 자체를 상품화합니다.
    *   **구체적 실행:** "갓더주식 예측 지표 보고서" 구독 모델을 도입하여, 실시간 매매 신호 대신 '강성 주식 포착 지표' 및 '리스크 경고 수준'을 제공하고 월별 구독료를 책정합니다.
    *   **예상 수익화 방식:** SaaS (구독 기반) 또는 정보 판매 (단일 상품).
    *   **근거:** 기술적 예측 실패는 시장의 변동성(Risk)이 증가했음을 의미하며, 사용자들은 불확실성에 대비하여 **정제된 정보**에 더 높은 가치를 부여합니다.

*   **B안 (리스크 헷지 서비스):** 알고리즘이 제시하는 '급등/강성주' 예측을 직접적인 매매 신호가 아닌, 포트폴리오 리스크 관리(Hedging) 도구로 재정의합니다.
    *   **구체적 실행:** 알고리즘 기반으로 생성된 **시장 위험 지수(Risk Index)**를 제공하고, 이를 활용하여 사용자에게 특정 자산군에 대한 헤지 전략을 제안하는 컨설팅 서비스를 개발합니다.
    *   **예상 수익화 방식:** B2B 리스크 관리 서비스 (고가 컨설팅).

#### 2. 리스크 헷지 전략 (Risk Hedging Strategy)

알고리즘의 예측 실패 시, 자본 보존을 최우선 목표로 설정하고 운영합니다.

*   **A안 (포트폴리오 분산):** 알고리즘이 집중하는 특정 섹터나 종목에 대한 의존도를 줄이고, **상관관계가 낮은 대체 자산(예: 채권, 금 등)**으로 포트폴리오를 분산시키는 전략을 실행합니다.
    *   **구체적 실행:** 예측 실패 시 자동으로 리스크 패널티를 발동하여 위험 자산을 축소하고 안정 자산 비중을 늘리는 자동 조정 모듈을 개발합니다.
    *   **근거:** 수익률(Return)보다 **자본 보존(Capital Preservation)**을 KPI의 최우선 기준으로 설정합니다.

*   **B안 (알고리즘 재보정):** 예측 실패 원인을 분석하여 모델 자체를 수정하는 데 자원을 집중합니다.
    *   **구체적 실행:** F2 지표 계산 모듈과 Feature Set 간의 상관관계(Correlation) 분석을 심화하여, 현재 시장 상황에 맞는 **가중치(Weighting)**를 재조정하고 백테스팅 루프를 반복 실행합니다.
    *   **근거:** 기술적 우위를 유지하기 위한 내부 자원 집중은 장기적인 경쟁력 확보에 필수적입니다.

### 🚀 최종 권고 (Decision & Rationale)

**최종 선택:** **A안 (정보 상품화) + B안 (알고리즘 재보정)의 병행 실행**

**이유:**
수익 예측 실패는 시스템의 기술적 오류(B안)와 비즈니스 모델의 적합성 문제(A안)를 동시에 시사합니다. 단순한 자본 헷지(A안)만으로는 지속 가능한 수익화가 불가능하며, 알고리즘 자체를 개선하지 않으면 시장에서 도태될 위험이 있습니다.

1.  **단기 (리스크 관리):** 예측 실패 시 즉각적으로 **포트폴리오 분산(A안)**을 발동하여 자본 손실을 최소화합니다.
2.  **중장기 (수익화 전환):** 알고리즘의 예측 정확도에 관계없이, 데이터 기반의 **'강성주 포착 지표'**라는 고유한 분석 결과물(A안)를 상품화하여 안정적인 수익 흐름을 창출하는 데 역량을 집중합니다.

### 🎯 1. 기술적 요구사항 (데이터 요구사항)

*   **A. 핵심 지표 연동 요구사항:**
    *   **TS (Time-Series):** 실시간 주가 및 거래량 데이터(`StockTimeSeriesData`)를 포트폴리오 구성 종목에 대해 **밀리초 단위로 업데이트**할 수 있어야 합니다. (F1 모듈의 실시간성 확보)
    *   **VSI/MSI (지표 기반):** 알고리즘이 산출하는 상승/급등/강성 주 지표(VSI, MSI)를 포트폴리오 리스크 측정 시 **정규화된 위험 점수**로 변환하여 제공해야 합니다.
    *   **KPI 연동:** 백테스팅 단계에서 도출된 **Sharpe Ratio 및 최대 낙폭(MDD)** 결과를 현재 포트폴리오의 실시간 성과 지표와 직접 비교할 수 있는 API 엔드포인트가 필요합니다.

*   **B. 모듈 기능 요구사항:**
    *   **리스크 계산 엔진:** 특정 자산군(섹터 또는 종목)에 대한 **상관관계 분석**을 수행하고, 예측 실패 시 포트폴리오의 **변동성 증가분($\Delta \sigma$)**을 실시간으로 산출해야 합니다.
    *   **헷지 전략 제안 모듈:** 시스템이 정의한 리스크 지표(예: $\text{MDD} > Y$ 또는 특정 섹터 내 상관관계 급변)가 발생했을 때, 사전에 정의된 **대체 자산군(채권, 금 등)**으로의 자동 전환 비율을 계산하여 제안해야 합니다.
    *   **알림 시스템:** 리스크 임계치 초과 시, 실시간으로 사용자에게 **리스크 경고 알림**을 전송하는 기능이 필수입니다.
