📱 영숙: 지시 확인했습니다. Writer와 Designer의 작업 진척도를 고려하여, 'API 연동 최종 점검 체크리스트'를 기반으로 실제 API 환경에서 데이터 권한 및 스키마 일치 여부를 검증하는 실행 테스트 계획을 구체화하겠습니다.

이 계획은 **실제 시스템 통합 테스트(Integration Test)** 단계에 초점을 맞추어, 구현된 로직의 정확성과 안정성을 확인하는 데 중점을 둡니다.

---

### 📋 API 연동 최종 점검 기반 실행 테스트 계획 (권한 및 스키마 검증)

**목표:** 키움 API 환경에서 데이터 접근 권한과 응답 스키마가 설계된 대로 완벽하게 일치하는지 검증합니다.

#### **1단계: 데이터 접근 권한 검증 (Authorization Test)**

이 단계는 시스템이 실제 API에 접근할 수 있는 '권한'을 확보했는지 확인합니다.

*   **테스트 1.1: 기본 인증 테스트**
    *   **실행:** Writer가 구현한 키움 API 연동 모듈(인증 부분)을 사용하여, 유효하지 않은 키 또는 만료된 토큰으로 요청을 시도합니다.
    *   **기대 결과:** 시스템이 `APIError` 또는 명확한 인증 오류 메시지를 반환해야 하며, 데이터가 노출되어서는 안 됩니다.
*   **테스트 1.2: 권한별 접근 테스트 (읽기/쓰기 분리)**
    *   **실행:** 설정된 API 키를 사용하여 '데이터 조회(Read)' 요청과 '데이터 수신(Stream)' 요청을 각각 시도합니다.
    *   **기대 결과:** 읽기 요청은 성공해야 하지만, 쓰기 또는 스트리밍 권한이 없는 엔드포인트에 대한 접근 시도는 반드시 거부되어야 합니다.
*   **테스트 1.3: 실시간 데이터 스트림 권한 확인**
    *   **실행:** 실시간 스트리밍(`start_stream`) 기능을 호출하여, 실제로 데이터 스트림을 받을 수 있는지 확인합니다.
    *   **기대 결과:** 스트림 연결이 성공적으로 유지되어야 하며, 연결 끊김이나 권한 오류가 발생하지 않아야 합니다.

#### **2단계: 스키마 일치성 검증 (Schema Consistency Test)**

이 단계는 API 응답 데이터의 구조(JSON Schema)가 Designer가 설계한 UI 컴포넌트와 완벽하게 일치하는지 확인합니다.

*   **테스트 2.1: 성공 케이스 스키마 비교**
    *   **실행:** API를 통해 정상적으로 데이터를 요청하고 응답을 수신합니다.
    *   **검증:** 수신된 JSON 객체의 필드 이름, 데이터 타입(숫자, 문자열, 리스트 등), 중첩 구조가 `writer.md`에서 정의한 **JSON Schema**와 100% 일치하는지 비교합니다. (특히 예측 지표: 상승/우상향/급등 예측 값)
*   **테스트 2.2: 예외 상태(Error State) 매핑 검증**
    *   **실행:** 테스트 1단계에서 의도적으로 오류 상황(예: 잘못된 파라미터 입력, 서버 오류 Mock)을 발생시킵니다.
    *   **검증:** API 응답에 포함된 에러 메시지나 상태 코드(`status_code`)가 Designer가 정의한 **Error State 디자인**에 따라 정확하게 매핑되는지 확인합니다. (예: '데이터 없음' 오류는 UI의 적절한 경고창으로 표시되어야 함)
*   **테스트 2.3: 데이터 타입 강제성 검증**
    *   **실행:** 예측 결과 값 중 숫자 필드에 문자열이 들어오거나, 리스트가 아닌 단일 값이 들어오는 비정상적인 데이터를 주입합니다.
    *   **검증:** 시스템이 입력값을 강제적으로 변환하거나 오류를 발생시켜 데이터 타입 불일치를 허용하지 않는지 확인합니다.

#### **3단계: 에러 핸들링 및 안정성 검증 (Robustness Test)**

구현된 예외 처리 로직의 안정성을 최종 점검합니다.

*   **테스트 3.1: Rate Limit 시뮬레이션**
    *   **실행:** 단시간 내에 과도한 요청을 보내어 API의 속도 제한(Rate Limit)이 작동하는지 확인합니다.
    *   **기대 결과:** 시스템이 서버 오류 대신 명확한 `RateLimitExceeded` 에러를 발생시키고, 이를 적절하게 사용자에게 알리는지 검증합니다.
*   **테스트 3.2: 비동기 처리 안정성**
    *   **실행:** 실시간 스트리밍(`DataStreamer` 스레드)이 장시간 구동될 때 메모리 누수나 프로세스 불안정성이 없는지 모니터링합니다.
    *   **기대 결과:** 시스템 리소스 사용량이 안정적으로 유지되어야 하며, 예상치 못한 크래시가 발생하지 않아야 합니다.
