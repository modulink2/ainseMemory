# 🧪 Phase 1: 알고리즘-API 통합 테스트 케이스 (Test Case Matrix)

**목표:** 자동 매매 예측 알고리즘이 실제 키움 API 데이터 스트림으로부터 안정적으로 데이터를 수집하고, 정의된 오류 시나리오 하에서 정확한 결과와 적절한 에러 핸들링을 수행하는지 검증한다.

**시스템 안정성 KPI 기준:**
*   **오류율 목표:** 0.5% 미만 (API 호출 및 데이터 처리 오류 포함)
*   **지연 시간 목표:** 평균 지연 시간 $\pm 5\%$

| **T1.4** | 이상치/노이즈 데이터 입력 | 극단적인 단일 시점의 비정상적 데이터 삽입 | 알고리즘이 노이즈를 필터링하고 안정적인 예측을 유지해야 함 | 성공 응답 (HTTP 200) | 안정성 및 노이즈 처리 능력 |

## 섹션 2: API 연동 및 스트리밍 안정성 검증 (API & Streaming Stability)

| ID | 테스트 시나리오 | 입력 데이터 조건 | 예상 결과 (API/스트림) | 예상 에러 핸들링 | 확인 항목 (KPI 연관) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **T2.1** | 실시간 데이터 스트리밍 | API로부터 5분 간격으로 연속적인 데이터 스트림 수신 | 데이터가 순서대로, 지연 없이(Latency $\pm5\%$) 수신되어야 함 | 성공 (데이터 정상 흐름) | 지연 시간 ($\pm5\%$), 데이터 무결성 |
| **T2.2** | API 연결 끊김 시뮬레이션 | 의도적으로 API 연결을 일시 중단/끊음 | 시스템은 비동기 방식으로 오류를 감지하고 재연결 메커니즘을 작동시켜야 함 | Actionable Feedback (API Down Error) 발생 후 자동 재시도 로직 실행 | 오류율 (0.5% 미만), 안정성 |
| **T2.3** | 데이터 스키마 불일치 | API 응답의 필드 이름이나 데이터 타입이 예상 JSON Schema와 다름 | 시스템은 데이터 매핑 실패를 즉시 포착하고, 정의된 Null/Default 값 처리 로직을 실행해야 함 | 데이터 스키마 불일치 오류 발생 및 로그 기록 | 오류율 (0.5% 미만), 데이터 무결성 |
| **T2.4** | 대량 데이터 부하 테스트 | 단시간 내에 최대 허용치 이상의 API 요청(스트림)을 집중 | 시스템 자원(CPU/메모리)이 안정적으로 유지되며, Latency가 허용 범위를 벗어나지 않아야 함 | 성공 (부하 처리 완료) | 지연 시간 ($\pm5\%$), 시스템 부하 내성 |

| :--- | :--- | :--- | :--- | :--- |
| **E3.1** | API 인증 실패 | 유효하지 않은 API 키 또는 만료된 토큰으로 요청 시도 | 즉시 인증 실패 오류를 반환하고, 재인증 프로세스를 요구해야 함 | "API 인증 실패: 토큰 만료" 알림 |
| **E3.2** | 데이터 권한 오류 | 접근 권한이 없는 종목 데이터에 대한 요청 시도 | 접근 거부(403 Forbidden) 또는 권한 부족 오류를 반환하고, 시스템은 해당 종목 데이터를 제외해야 함 | "데이터 접근 권한 부족" 로그 기록 |
| **E3.3** | 예측 알고리즘 계산 실패 | 입력 데이터가 너무 불안정하여 알고리즘이 수렴하지 못하는 경우 (NaN 또는 무한대 발생) | 알고리즘은 오류 상태를 반환하고, 자동 매매 실행을 중단해야 함 | "알고리즘 수렴 실패: 안정성 경고" 알림 및 매매 중단 |
| **E3.4** | 시스템 메모리 부족 | 실시간 데이터 스트리밍 시점에 메모리 할당에 실패하는 상황 발생 | 시스템은 예외를 포착하고, 데이터를 일시 중단하거나 작은 모델로 전환하는 안전 모드로 전환해야 함 | "시스템 리소스 부족: 작업 일시 중단" 경고 |
