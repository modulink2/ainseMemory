## 📊 최종 디자인 컨셉 확정: 신뢰 기반 대시보드 (Trust-Based Dashboard)

### 1. 비주얼 컨셉 및 브랜드 연관성
**핵심 주제:** 데이터의 투명성과 시스템의 안정성이 곧 투자 신뢰라는 메시지 전달.
**톤앤매너:** 차갑고, 정밀하며, 절대적인 확신을 주는 **'기술적 권위(Technical Authority)'** 톤. (갓더주식의 자가학습 기반 기술력을 반영)

| 디자인 요소 | 적용 컨셉 | 시각적 목표 | 브랜드 연관성 |
| :--- | :--- | :--- | :--- |
| **컬러 팔레트** | Deep Navy (주요 배경), Electric Blue/Cyan (핵심 지표), Subtle Gray (보조 정보) | 데이터의 중요성과 기술적 정밀함을 강조. 신뢰감을 주는 차가운 톤 유지. | 기술적 안정성(KPI)을 시각화하여 '감'이 아닌 '증명'을 제공. |
| **타이포그래피** | Monospace 또는 Sans-serif (예: Inter, Fira Code 계열) | 데이터의 정확성과 가독성을 극대화. 수치 강조에 최적화. | 시스템 안정성 KPI를 명확하게 전달하여 정보 과부하 방지. |
| **레이아웃 구조** | Dashboard Kit 기반의 **'신뢰 흐름(Trust Flow)'** 구조 채택. (상단: 요약/안정성, 중앙: 예측 결과, 하단: 상세 로그/증명) | 사용자가 가장 먼저 시스템의 안정성을 확인하고, 그 다음 예측 결과를 보게 유도하여 신뢰를 선행 구축. |

### 2. 핵심 UI/UX 흐름 (Trust Flow Mapping)

대시보드 화면은 세 단계의 명확한 신뢰 흐름을 따라 설계되어야 합니다.

#### **Stage 1: 안정성 확인 (The Foundation of Trust)**
*   **위치:** 대시보드의 최상단, 가장 눈에 띄는 영역 (Header/Top Bar).
*   **표시 항목:** 시스템 핵심 KPI를 게이지(Gauge) 또는 바(Bar) 형태로 표시.
    *   **Primary KPI:** **시스템 안정성 지표 (오류율 $\le 0.5\%$, 지연 시간 $\pm 5\%$)**. 이 지표는 녹색/빨간색이 아닌, **'안정 상태(Stable State)'**를 나타내는 청록색 또는 밝은 파란색으로 표시하여 긍정적인 확신을 주어야 합니다.
    *   **보조 KPI:** 실시간 데이터 연결 상태 (API Latency Status).

#### **Stage 2: 예측 결과 (The Insight)**
*   **위치:** 대시보드의 중앙 영역.
*   **표시 항목:** 알고리즘의 최종 예측값 및 확률.
    *   예측 결과(Up/Down/Neutral)를 가장 크고 명확하게 표시하며, 예측에 대한 **신뢰도 점수(Confidence Score)**를 함께 제시합니다. (이는 Writer가 제시한 '자신감'을 시각화.)

#### **Stage 3: 증명 및 관리 (The Proof & Control)**
*   **위치:** 중앙 영역 하단 또는 사이드바.
*   **표시 항목:** 시스템의 투명성을 확보하는 상세 데이터 및 로그 접근 경로.
    *   **Error State 통합:** Writer가 설계한 **'최종 Error State 디자인'**을 이 영역에 적용하여, 오류 발생 시 사용자가 즉각적으로 원인과 해결책(시스템 신뢰도 관점)을 이해하도록 유도합니다.
    *   **Audit Log Link:** 모든 예측 및 매매 결정에 대한 간략한 감사 로그 접근 버튼 제공.

### 3. Phase 2 개발 우선순위 (Design Specification for Development)

확정된 비주얼 컨셉을 기반으로, 다음 단계의 개발 우선순위를 다음과 같이 설정합니다. 이 순서는 **'신뢰 확보 $\rightarrow$ 예측 시각화 $\rightarrow$ 시스템 검증'**의 논리적 흐름을 따릅니다.

#### **Priority 1: 안정성 KPI 시각화 모듈 완성 (Foundation)**
*   **목표:** Stage 1(안정성 확인) 구현.
*   **디자인 사양:**
    *   KPI 게이지는 선형 진행이 아닌, **'실시간 상태 표시등(Status Indicator)'** 형태로 디자인한다. (예: 오류율 0.5% 미만 시 녹색 불빛 지속)
    *   오류 발생 시, 해당 KPI 영역 전체가 경고 색상으로 즉시 전환되도록 컴포넌트 정의.
*   **개발 요구사항:** 데이터 수신 모듈에서 나오는 오류율/지연 시간 값을 받아 이 상태 표시등을 실시간으로 업데이트하는 API 연동 로직 설계.

#### **Priority 2: 예측 결과 시각화 및 신뢰도 레이어 추가 (Insight)**
*   **목표:** Stage 2(예측 결과) 구현.
*   **디자인 사양:**
    *   예측값은 중앙에 배치하되, 예측 주변에 **'신뢰도 버블(Confidence Bubble)'**을 삽입하여, 예측 수치($X$)와 신뢰도 점수($Y$)가 분리되어 시각적으로 명확히 대비되도록 설계. (Writer의 '자신감'을 반영)
    *   색상 사용: 예측 결과는 강조 색상(Cyan/Blue), 신뢰도는 보조 색상(Gray scale).
*   **개발 요구사항:** 알고리즘 출력에 따라 Confidence Score를 계산하고, 이 점수를 기반으로 텍스트 및 시각적 요소의 크기를 동적으로 조정하는 프론트엔드 로직 구현.

#### **Priority 3: 예외 상태 통합 UI/UX (Control)**
*   **목표:** Stage 3(증명 및 관리) 구현.
*   **디자인 사양:**
    *   Error State는 단순한 에러 메시지가 아닌, **'시스템 신뢰도에 대한 진단 리포트'** 형태로 디자인한다. (예: "오류율이 임계치($0.5\%$)를 초과했습니다. 원인 분석을 시작하세요.")
    *   사용자는 오류 발생 시 단순히 재시작 버튼을 누르는 것이 아니라, **시스템 상태(KPI)에 대한 진단**을 먼저 요구받도록 설계하여 능동적인 관리자로 포지셔닝한다.
*   **개발 요구사항:** 모든 예외 처리 경로(API 실패, 데이터 불일치 등)에서 사용자에게 제시할 표준화된 진단 템플릿 정의 및 해당 템플릿에 따른 UI 컴포넌트 매핑.
