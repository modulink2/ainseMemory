### 📊 KPI 구조 최종 검토 및 백엔드 요구사항 정의

#### 1. KPI 구조 최종 검토 (기존 로직 기반)

기존의 시스템 안정성 KPI와 수익화 KPI를 다음과 같이 통합하여 구독 티어별 차별점을 명확히 설정합니다.

| KPI 카테고리 | 세부 지표 | 측정 대상 | Basic Shield ($19) | Pro Hedger ($49) | 비즈니스 의미 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **시스템 안정성 (Safety)** | $MDD_{limit}$ 준수율 | 시스템 안전 모드 활성화 횟수/시간 | 기준치(예: 95%) 유지 | 강화된 실시간 모니터링 및 사전 경고 기능 포함 | 리스크 회피 능력의 차이 |
| **수익성 (Profitability)** | 헤징 ROI | 실제 헤징을 통해 발생한 수익/손실 | 기본 헤징만 제공 | 고급 예측 알고리즘 접근 및 자동 최적화 제공 | 투자 기회의 질 차이 |
| **사용자 경험 (UX/Access)** | API 호출 제한 | 사용자별 API 사용 횟수/빈도 | 제한된 횟수 | 무제한 또는 고빈도 데이터 스트리밍 | 기능 접근성 |

**핵심 결론:** Basic 티어는 '안정성 확보'에 중점을 두고, Pro 티어는 '수익 극대화 기회 포착'에 중점을 두도록 KPI 가중치를 설정합니다.

#### 2. 백엔드 요구사항 정의 (데이터 전달 및 UI/UX 연결)

이러한 정량화된 KPI가 사용자에게 명확히 전달되고 구독 결정에 영향을 미치기 위해 다음의 백엔드 및 데이터 흐름을 요구합니다.

**A. 실시간 KPI 스트리밍 모듈 요구사항:**
1.  **KPI 집계 엔드포인트 정의:** 모든 시스템 상태(안정성, 수익성)를 실시간으로 집계하여 사용자 세션에 제공하는 REST API 또는 WebSocket 엔드포인트를 정의해야 합니다. (예: `/api/v1/user/kpi_status`)
2.  **티어별 필터링 로직:** 백엔드는 사용자가 구독한 티어(Basic/Pro)를 기준으로, 수집된 KPI 데이터 중 해당 티어에 **관련 있는 지표만** 선별하여 반환하는 필터링 로직을 구현해야 합니다. (예: Basic 사용자에게는 $MDD_{limit}$ 준수율과 기본 헤징 ROI만 제공).

**B. 가치 정량화 대시보드 요구사항:**
1.  **가치 점수(Value Score) 산출 알고리즘:** 시스템 안정성 점수와 수익성 점수를 결합하여 사용자가 직관적으로 이해할 수 있는 단일 **'플랫폼 신뢰 지수(Platform Trust Index, PTI)'**를 산출하는 로직을 정의하고 이를 백엔드에서 계산하도록 요구합니다.
    *   $PTI = w_1 \times (\text{안정성 점수}) + w_2 \times (\text{수익성 점수})$ (여기서 $w_1, w_2$는 티어별로 차등 적용)
2.  **데이터 시각화 레이어 요구사항:** 이 산출된 PTI와 각 세부 KPI가 Designer가 정의한 Deep Navy Blue & System Green 디자인 가이드라인에 맞춰 시각적으로 표현될 수 있도록 데이터 포맷(JSON 스키마)을 명확히 정의해야 합니다.

**C. 데이터 무결성 및 신뢰 보장 요구사항:**
1.  **데이터 출처 명시:** 모든 KPI 데이터가 어떤 모듈(`risk_hedge_integration.py`, `system_safe_mode.py` 등)에서 왔는지에 대한 메타데이터를 API 응답에 포함하여, 사용자가 지표의 신뢰도를 즉각적으로 판단할 수 있도록 해야 합니다.
