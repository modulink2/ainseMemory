### 1. 기술적 구현 가능성 검토 (Feasibility Check)

**결론:** 제시된 시각적 요구사항은 **기술적으로 완벽하게 가능하며, 오히려 시스템 안정성을 최우선으로 하는 플랫폼 철학에 가장 잘 부합합니다.**

*   **PTI 우선 배치:** PTI 지표를 최상단(A 영역)에 System Green으로 강조하는 것은 데이터의 계층 구조를 시각적으로 정의하는 명확한 방법입니다. 이는 사용자가 정보를 처리할 때 **가장 중요한 정보(안정성)**를 즉시 인지하게 하여, 리스크 제어형 플랫폼이라는 핵심 가치를 전달합니다.
*   **색상 적용 (Deep Navy Blue & System Green):** 이 색상 조합은 '신뢰(Navy)'와 '안전/수익(Green)'이라는 핵심 철학을 시각적으로 일관되게 연결합니다. PTI가 녹색일 때 시스템이 안전함을, 위험할 때 붉은색 경고를 통해 즉각적인 액션 트리거를 제공하는 UX 흐름을 구축하기에 최적화되어 있습니다.
*   **데이터 흐름 안정성:** 핵심은 시각화(Presentation Layer)가 데이터 소스(Data Layer)의 무결성에 직접적으로 종속되어야 한다는 점입니다. PTI 계산 모듈에서 도출된 값이 UI에 반영되는 과정이 단 한 번의 오류도 없이 **실시간으로 동기화**되도록 백엔드 로직(`risk_hedge_integration.py` 등)의 안정성이 UI 구현의 전제 조건이 됩니다.

### 2. 핵심 KPI 배치 및 데이터 흐름 초기 구조 설계 의견 (Structural Blueprint)

시스템 안정성(PTI)을 중심으로 데이터를 계층적으로 구성하는 아키텍처를 제안합니다.

#### A. 데이터 흐름 원칙: Bottom-Up & Trust-Driven
모든 시각화는 **PTI 계산 모듈의 결과**에서 시작되어야 하며, 모든 KPI는 PTI에 대한 종속성을 명확히 보여줘야 합니다.

1.  **Base Layer (Source of Truth):** 실시간 시장 데이터, API 응답, 시스템 상태 로그 (`data_stream`)
2.  **Processing Layer (Core Logic):** **PTI 계산 모듈** 및 리스크 헷지 로직 (`risk_hedge_integration.py`). 이 단계가 모든 결과의 신뢰도를 결정합니다.
3.  **Presentation Layer (UI/UX):** Designer가 설계한 레이아웃에 따라, Processing Layer의 결과를 시각화하여 표시합니다.

#### B. 대시보드 구조 상세 제안 (Layout Refinement)

| 영역 | 구성 요소 | 데이터 소스 및 로직 연관성 | 디자인 강조점 |
| :--- | :--- | :--- | :--- |
| **A. 최상단 헤더** | **PTI 지표 (가장 크게)**, 시스템 상태 (`System Safe Mode`) | Processing Layer의 최종 결과 (시스템 안정성 지수) | **System Green**. 실시간 변화를 가장 먼저 인지시킴. |
| **B. 안전 모드 제어** | 자동매매 허용/정지 토글 버튼 | `system_safe_mode` 상태 변수 | 명확하고 즉각적인 조작 가능성 제공. (위험 시 강제 알림) |
| **C. 핵심 지표 섹션** | ROI, 포트폴리오 가치, 최근 수익률 변화 | Processing Layer의 결과 및 Base Layer 데이터 | PTI 대비 보조적인 역할. 안정성이 확보되었을 때만 수익 지표를 강조하도록 로직 구성. |
| **D. 시스템 안정성 시각화** | PTI 추이 그래프 (Time Series), 위험 알림 영역 | Processing Layer의 히스토리 데이터 | **PTI 변화 추이를 중심으로**. 녹색/빨간색 경고 영역(Zone)을 통해 리스크 구역 명시. |
| **E. 포트폴리오 상세** | 개별 종목 현황, 리스크 헷지 상태 | Base Layer 및 Processing Layer의 상세 계산 결과 | 세부 데이터 제공. (상위 지표에 대한 근거 제시). |

#### C. 초기 구조 설계 의견 (Actionable Steps)

1.  **API 스키마 통일:** Designer가 요구하는 모든 KPI(`PTI`, `ROI`, 위험도)가 백엔드 API에서 일관된 포맷으로 반환되도록 `api_schema_definition.py`를 재검토하고, 이 스키마가 UI에 반영되는 **단일 진실의 원천(Single Source of Truth)**이 되도록 확정해야 합니다.
2.  **동적 색상 로직:** PTI 값에 따라 색상이 동적으로 변화하는 로직(`PTI > Threshold` 일 때 Red 경고 활성화)은 프론트엔드에서 구현하되, 이 **Threshold 값의 설정 기준**을 백엔드의 `risk_hedge_integration.py`와 완벽히 일치시켜야 합니다.
3.  **안정성 우선 로직:** 모든 데이터 업데이트는 시스템 안정성 지표(`PTI`)를 최우선으로 검증하는 필터를 거쳐야 합니다. 만약 데이터 무결성 실패가 감지되면, UI상에 경고 메시지가 **시스템 안전 모드(B 영역)**와 연동되어 즉시 사용자에게 조치를 요구해야 합니다.

이 구조대로라면, 시각적 아름다움뿐만 아니라 시스템의 핵심 목표인 '안정성'을 모든 화면에서 최우선으로 인식하게 만들 수 있습니다.
